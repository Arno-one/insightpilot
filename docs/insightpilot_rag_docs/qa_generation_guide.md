# InsightPilot QA 问答对制作建议

doc_id: qa_generation_guide_v1
适用场景: 将原始 RAG 文档加工成 QA 问答对
文档版本: v1.0

## 为什么要做 QA 问答对

原始文档适合覆盖完整知识，但用户真实提问往往很短，比如“客户嫌贵怎么办”“报价后不回复怎么办”。如果只检索原始文档，可能需要依赖向量相似度命中。补充 QA 问答对后，可以提高高频问题的命中稳定性。

建议策略：

- 原始文档负责完整上下文。
- QA 问答对负责高频问题精确命中。
- Agent 最终回答时可以同时引用 QA 和原始文档。

## 推荐 QA 字段

建议每条 QA 至少包含：

```json
{
  "id": "qa_obj_001_001",
  "question": "客户说价格太贵怎么办？",
  "answer": "先不要马上降价，应确认客户觉得贵的原因，是预算有限还是价值没有被看见。可以回应：我理解您会关注预算，我们先确认一下，您觉得价格高主要是因为总预算有限，还是还没有完全看到系统能减少哪些销售管理损失？",
  "doc_id": "objection_handling_v1",
  "section_id": "OBJ-001",
  "tags": ["异议处理", "价格太贵", "报价阶段", "销售话术"],
  "source_type": "qa"
}
```

## 每个知识点建议生成的问题数量

每个小节建议生成 5-10 个问题变体。

示例：

- 客户说价格太贵怎么办？
- 客户嫌贵怎么回复？
- 报价后客户觉得预算不够怎么办？
- 竞品比我们便宜怎么说？
- 销售员能不能直接给客户打折？

这些问题都可以指向 `OBJ-001 客户说价格太贵`。

## 推荐 QA 类型

### 事实型问题

用于查询规则、定义和标准。

示例：

- 高风险客户的分数范围是多少？
- 报价后多久不回复算风险？
- 经营日报必须包含哪些内容？

### 操作型问题

用于指导销售下一步动作。

示例：

- 客户报价后 14 天没回复应该怎么处理？
- 高风险客户是否需要主管介入？
- AI 生成任务后能不能自动创建？

### 话术型问题

用于生成具体沟通文本。

示例：

- 客户说没预算怎么回复？
- 客户说要再考虑怎么跟进？
- 客户担心 AI 不可靠怎么解释？

### 权限型问题

用于解释角色边界和安全机制。

示例：

- 销售员能看全公司客户风险吗？
- 老板和主管看到的数据有什么不同？
- Agent 调用工具前是否检查权限？

## 负例和边界问题也要做

为了避免 Agent 乱答，建议补充一些边界问题。

示例：

- AI 能不能跳过主管确认直接创建任务？
- 销售员能不能审批自己的 AI 任务？
- 客户只比较价格时是否应该直接降价？
- 客户没有需求也没有预算时是否继续高频跟进？

推荐答案要明确限制：

- AI 不能跳过人工确认直接创建正式任务。
- 销售员不能审批自己负责客户的 AI 任务草稿。
- 不建议无条件降价，应先确认客户对价值的理解。

## 文件组织建议

建议你后续可以生成：

- `qa_sales_sop.jsonl`
- `qa_product_pricing.jsonl`
- `qa_objection_handling.jsonl`

每行一条 JSON，方便批量入库。

JSONL 示例：

```jsonl
{"id":"qa_sop_003_001","question":"高风险客户是多少分？","answer":"高风险客户的风险分范围是 70-100，表示客户存在明显流失信号，可能已转向竞品、预算受限或长时间未回应。","doc_id":"sales_sop_v1","section_id":"SOP-003","tags":["风险等级","高风险客户"],"source_type":"qa"}
{"id":"qa_obj_005_001","question":"报价后客户一直不回复怎么办？","answer":"报价后长时间不回复是客户流失的核心信号。建议先用低压力方式重新建立连接，不要重复催促，并提供一个容易回应的问题。高金额客户多次无回应时建议主管介入。","doc_id":"objection_handling_v1","section_id":"OBJ-005","tags":["报价跟进","客户无回应","流失风险"],"source_type":"qa"}
```

## 入库建议

建议给 QA 和原始文档设置不同的 `source_type`。

- 原始 Markdown：`source_type=document`
- QA 问答对：`source_type=qa`

检索时可以先查 `qa`，如果 QA 分数高于阈值，直接把 QA 答案作为强上下文；如果 QA 分数不足，再查 `document` 获取完整解释。
