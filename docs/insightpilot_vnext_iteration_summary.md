# InsightPilot 下个版本迭代功能与模块总览

版本日期：2026-07-01

## 1. 文档目的

这份文档只回答四个问题：

- 当前版本已经落地了哪些关键能力。
- 下个版本不应该再重复补哪些已经完成的内容。
- 下个版本应该优先迭代哪些功能和模块。
- 每个模块推进时，最少要验收到什么程度才算完成。

## 2. 当前版本已完成的基线

结合现有代码、数据库和最近几轮迭代结果，当前版本已经不是纯骨架，而是一个可演示、可联调、可回归的 V1 Demo，已具备以下基线能力：

### 2.1 基础工程与系统骨架

- 后端已完成 `FastAPI + SQLAlchemy + Redis + RQ + APScheduler` 基础工程搭建。
- 前端已完成 `Next.js` 工作台结构和主导航页面接入。
- MySQL、Milvus、OpenAI/DeepSeek、Embedding、Worker 调度链路已接通。
- 前后端基础登录态、JWT 鉴权、RBAC 权限校验已跑通。

### 2.2 数据库与演示数据

- 数据库表结构已重建。
- 表和字段中文注释已补齐。
- 当前库中已补入一批更接近真实业务的模拟数据。
- 已明确采用“不设数据库外键，只通过同名字段关联”的建模约束。
- Alembic baseline 已生成，后续可以继续做增量迁移。

### 2.3 核心业务闭环

- CRM 客户、商机、跟进记录可查询。
- 风险规则识别已落地，能生成风险快照。
- RAG 入库、检索、评估、Trace 已接通。
- AI 风险建议已接入人工审批流程。
- 审批通过后可创建正式销售任务。
- 任务支持开始执行、完成、取消，并可回写跟进记录。
- 风险状态、审批、任务、日报、Trace 已完成一轮完整回归联调。

### 2.4 Agent 与可观测性

- 风险扫描已切到 `LangGraph` 图编排。
- 经营日报已切到 `LangGraph` 图编排。
- Agent Run、Agent Step、RAG Trace 已具备前后端查看能力。
- 已补过一轮“编码安全回归”与测试文档。

### 2.5 系统管理与展示层优化

- 已新增 `admin` 角色。
- 系统管理页已具备角色权限开关与用户角色分配能力。
- 多处“审批人 / 发起人 / 负责人 / 当前席位 / 报告归属人”展示已统一朝 `real_name` 对齐。
- 前端新增了隔离构建校验，避免 `next dev` 运行时被 `build` 污染。

## 3. 下个版本的核心目标

下个版本不再以“补骨架”为目标，而是把系统从“可演示”推进到“更像真实可用的业务工作台”。

建议把目标收敛为四句话：

- 让真实业务数据更容易进入系统。
- 让客户风险到执行动作的闭环更完整。
- 让报表、客户视图和管理视图更适合日常使用。
- 让 LangGraph、权限和回归体系更稳，便于继续扩展。

## 4. 下个版本建议优先迭代的模块

## 4.1 模块一：数据接入与数据治理

### 目标

把系统从“依赖演示库”推进到“可以导入一批外部 CRM 数据做真实演示”。

### 建议迭代功能

- 新增 CRM 数据 `CSV` 导入能力。
- 支持客户、商机、跟进记录三类核心数据导入。
- 导入前做字段映射、必填校验、格式校验、重复数据校验。
- 导入后记录导入批次、导入结果、失败原因和操作人。
- 支持下载失败数据清单，方便二次修正。

### 最低验收标准

- 用户能上传一个标准模板并成功导入客户数据。
- 导入失败时能看到明确错误原因，而不是只返回失败。
- 导入的数据能立即在客户、风险、任务相关页面被查询到。

## 4.2 模块二：客户详情页与风险处置工作台

### 目标

把“风险中心列表”升级成“客户级处置入口”，让主管和销售能围绕一个客户连续处理问题。

### 建议迭代功能

- 新增客户详情页。
- 展示客户基础信息、负责人、生命周期阶段、最近跟进、最近商机、历史风险快照。
- 展示某次风险被识别出来的原因、证据、建议动作和推荐话术。
- 支持从客户详情页直接触发风险重算或重新生成建议。
- 支持查看该客户关联的审批、任务、跟进记录和日报引用情况。

### 最低验收标准

- 从风险中心能点进客户详情。
- 客户详情页能看清“为什么高风险、谁负责、做过什么、下一步做什么”。
- 风险重算后，页面能看到新的风险快照或新的建议结果。

## 4.3 模块三：审批与任务执行闭环增强

### 目标

把当前“最小执行闭环”升级成“更可管理、更可追责、更可复盘”的执行系统。

### 建议迭代功能

- 审批列表支持更多筛选条件，如状态、审批人、发起人、时间范围。
- 任务列表支持负责人、优先级、状态、逾期情况筛选。
- 增加任务操作历史和审批操作历史。
- 增加批量审批、批量分配、批量更新任务状态能力。
- 对任务取消、审批驳回、再次提交形成完整审计记录。
- 根据任务完成结果，支持自动触发客户风险重算或重新生成日报素材。
- 继续细化任务权限，例如“仅看本人 / 看团队 / 看全部”在操作按钮层也保持一致。

### 最低验收标准

- 一条建议从审批到任务到跟进再到风险更新，链路上每一步都能追溯。
- 管理角色能快速筛出待处理任务和逾期任务。
- 关键操作都能留痕，方便后续审计或回放。

## 4.4 模块四：经营报告升级

### 目标

让报告不只停留在“日报列表”，而是升级成管理层可以持续消费的经营视图。

### 建议迭代功能

- 在现有日报基础上增加周报、月报。
- 增加趋势指标，例如风险客户变化、待审批积压、任务完成率、逾期率。
- 报告中按归属人展示数据，明确是哪个销售、哪个主管、哪个团队的问题或成果。
- 支持报告按时间范围筛选。
- 支持从报告跳转到风险、任务、审批明细。
- 预留导出能力，例如导出为文本摘要或结构化数据。

### 最低验收标准

- 日报、周报、月报三类报告都能生成并正常展示。
- 报告里不只是堆列表，还能看出趋势和责任归属。
- 报告点击后能继续钻取到具体业务对象。

## 4.5 模块五：系统管理与权限治理增强

### 目标

把已有的 `admin + 权限开关` 从“演示可用”推进到“规则更完整、边界更清晰”。

### 建议迭代功能

- 权限管理页增加按模块分组展示权限点。
- 增加角色权限差异对比能力。
- 增加用户数据范围配置，例如“仅本人 / 本团队 / 全部”。
- 增加权限变更日志和角色分配日志。
- 增加系统初始化角色模板能力，降低新环境搭建成本。

### 最低验收标准

- `admin` 能清楚看到每个角色拥有的功能权限。
- 权限变更后，前端菜单、按钮、后端接口访问结果保持一致。
- 权限调整过程可追溯。

## 4.6 模块六：LangGraph 与 Agent 工程治理

### 目标

既然两条核心链路已经切到 `LangGraph`，下个版本就要开始做图编排层的可维护性治理。

### 建议迭代功能

- 继续拆分图内 `state / node / repository / service` 责任边界。
- 统一图节点输入输出结构，减少隐式字段耦合。
- 增加节点级超时、重试、幂等保护和失败告警。
- 让 Agent Trace 展示更贴近真实节点语义，而不只是技术日志。
- 为风险扫描图和日报图补齐更明确的测试夹具和回归样例。

### 最低验收标准

- 图节点代码不再继续膨胀成“大文件黑盒”。
- 节点失败时能明确定位失败阶段和失败原因。
- Trace 能帮助排查问题，而不是只能证明“跑过”。

## 4.7 模块七：回归测试与稳定性建设

### 目标

把这套系统从“能修问题”推进到“后续改动不容易把老功能改坏”。

### 建议迭代功能

- 建立固定的全链路回归清单：审批 -> 任务 -> 跟进 -> 风险状态 -> 日报 -> Trace。
- 建立角色维度回归清单：`owner / manager / salesperson / admin`。
- 建立前端关键页面稳定性检查，尤其是登录态、用户展示、列表页和报告页。
- 建立开发态构建规范，避免再次出现 `.next` 构建污染问题。
- 在 `docs/testing` 继续沉淀每轮测试记录和已知风险说明。

### 最低验收标准

- 每次发版前至少能执行一轮固定脚本或固定清单回归。
- 已修复的问题有对应回归项，避免重复出现。
- 文档里能清楚说明“怎么测、测什么、哪些地方最容易坏”。

## 5. 建议的版本优先级

如果下个版本只做一轮聚焦迭代，建议优先级按下面顺序推进：

### P0：必须优先做

- 数据接入与数据治理
- 客户详情页与风险处置工作台
- 审批与任务执行闭环增强

### P1：建议本版本完成

- 经营报告升级
- 系统管理与权限治理增强

### P2：建议并行持续推进

- LangGraph 与 Agent 工程治理
- 回归测试与稳定性建设

## 6. 推荐的迭代拆分方式

为了避免下个版本再次变成“大而全”，建议拆成三个阶段：

### 阶段一：先补业务入口

- 做 CSV 导入
- 做客户详情页
- 打通风险列表到客户详情的跳转

### 阶段二：再补执行闭环

- 做审批和任务的筛选、批量操作、历史留痕
- 做任务结果反推风险状态

### 阶段三：最后补管理和治理

- 做周报、月报和趋势分析
- 做权限治理增强
- 做 LangGraph 工程治理和标准化回归清单

## 7. 一句话结论

下个版本最值得投入的方向，不是继续补零散页面，而是围绕“数据进来 -> 风险识别 -> 审批决策 -> 任务执行 -> 经营复盘”这条主链路，把客户详情、数据导入、执行闭环和管理报表做深做实，这样 InsightPilot 才会从一个演示系统，继续向真实业务工作台靠近。
