# InsightPilot 当前版本功能清单与下一版本迭代路线

版本日期：2026-07-01

## 1. 文档目的

本文用于记录 InsightPilot 当前已实现的功能模块、已知问题、未完成能力，以及下一版本建议迭代方向。

当前项目已经具备一个可演示的 V1 主链路，但仍存在前端乱码、页面排版、真实 LangGraph 编排、CSV 导入、Alembic 迁移等需要继续完善的部分。本文作为后续开发的迭代基准。

## 2. 当前版本定位

InsightPilot 当前版本是一个面向中小企业销售运营场景的 AI 企业运营参谋系统。

核心主线是：

```text
CRM 数据
  -> 规则引擎识别客户风险
  -> RAG 检索销售 SOP / 产品资料 / 异议处理话术
  -> LLM 生成风险解释和处理建议
  -> 人工审批 AI 任务草稿
  -> 创建正式销售任务
  -> 生成经营日报
  -> 通过 Agent Trace 追踪执行链路
```

当前版本更偏“可演示、可解释、可扩展”的 V1，不是最终完整 SaaS 产品。

## 4. 当前版本已知问题

### 4.1 前端存在乱码问题

当前前端页面和部分文档中仍可看到 `???` 或类似 `浠诲姟` 这样的乱码。

可能原因：

- 早期文件在 PowerShell 或编辑器中被错误编码保存。
- 部分中文内容在终端输出时使用了不兼容编码。
- 旧文档或旧页面在创建时不是标准 UTF-8。

影响：

- 页面展示不专业。
- 会影响项目演示观感。
- 会降低 HR 或面试官对项目完成度的第一印象。

下个版本必须将“全项目中文编码修复”列为前端和文档优化重点。

### 4.2 前端排版仍需优化

当前前端已经能跑通真实数据，但还存在以下问题：

- 表格列较多时移动端体验一般。
- 风险原因、建议、推荐话术等长文本展示不够精致。
- 驾驶舱的数据层次还不够强。
- 审批台和 Agent Trace 页面可读性还可以继续提升。
- 部分按钮、卡片、间距、状态标签需要更统一的设计系统。

### 4.3 LangGraph 尚未真正落地

当前风险扫描和经营日报是 Worker 内部顺序执行，并未真正用 LangGraph 编排。

目前已有占位文件：

- `backend/app/modules/agent/graphs/risk_analysis_graph.py`
- `backend/app/modules/agent/graphs/business_report_graph.py`

下个版本应把当前顺序流程迁移为真实 LangGraph 节点。

### 4.4 CSV 导入未实现

用户最初定过“先内置模拟数据，再支持 CSV 导入”。

当前只完成了内置模拟数据，尚未支持：

- 客户 CSV 导入。
- 联系人 CSV 导入。
- 商机 CSV 导入。
- 跟进记录 CSV 导入。
- 导入校验和错误报告。

### 4.5 Alembic 迁移未完善

当前已有 Alembic 骨架和初始化 SQL，但还没有完整 migration revision。

后续应补：

- 首个正式 migration。
- 与初始化 SQL 的一致性检查。
- 本地重建数据库说明。

### 4.6 真实 Docker Milvus 全链路还需复测

RAG 功能主要用 Milvus Lite 完成测试。

当前 Docker Milvus 已能启动，但建议下个版本重新跑：

- Docker Milvus 入库。
- Docker Milvus 检索。
- Docker Milvus RAG 评估。
- RAG Trace 写入校验。

### 4.7 RQ 真实端到端链路需要补测

Redis 已启动，RQ Worker 已能监听队列，但仍建议做一次完整按钮触发测试：

- 前端点击风险扫描。
- Worker 消费风险扫描任务。
- 前端查看风险快照更新。
- 前端点击经营日报生成。
- Worker 消费日报任务。
- 前端查看最新日报。
- 前端触发 RAG 入库。
- Worker 消费 RAG 入库任务。

### 4.8 任务执行闭环还不完整

当前任务主要是“审批通过后创建”和“列表查看”。

尚未完整实现：

- 任务开始处理。
- 任务完成。
- 任务取消。
- 任务逾期自动标记。
- 填写执行结果。
- 根据任务结果反向更新客户状态。

## 5. 下一版本建议迭代模块

### 5.1 前端乱码与 UI 排版专项优化

优先级：最高。

目标：

- 修复所有前端页面乱码。
- 修复文档中关键中文乱码。
- 优化页面布局、表格、卡片、状态标签和长文本展示。
- 形成统一设计系统。

建议任务：

- 全量扫描 `.tsx`、`.ts`、`.css`、`.md` 文件中的乱码。
- 统一保存为 UTF-8。
- 优化以下页面：
  - 登录页。
  - 驾驶舱。
  - 风险中心。
  - 审批台。
  - 销售任务。
  - 经营报告。
  - Agent Trace。
  - RAG 评估。
- 增加空状态、错误状态、加载状态和成功提示的一致设计。

验收标准：

- 页面不再出现乱码。
- 中文内容自然、专业。
- 移动端和桌面端排版都可用。
- HR 或面试展示时不需要解释“这里为什么有乱码”。

### 5.2 真实 LangGraph 编排

优先级：最高。

目标：

- 将风险扫描流程迁移到 LangGraph。
- 将经营日报流程迁移到 LangGraph。
- 让 Agent Trace 中的节点和 LangGraph 节点保持一致。

风险分析图建议节点：

```text
load_crm_data
  -> calculate_rule_risk
  -> retrieve_sales_knowledge
  -> generate_risk_reason
  -> create_approval_record
  -> persist_agent_trace
```

经营日报图建议节点：

```text
collect_business_metrics
  -> analyze_risk_top
  -> generate_report_narrative
  -> persist_business_report
  -> persist_agent_trace
```

验收标准：

- Worker 调用 LangGraph graph，而不是手写顺序流程。
- Agent Step 与 LangGraph 节点一致。
- 出错节点可追踪。

### 5.3 CSV 导入模块

优先级：高。

目标：

- 支持从 CSV 导入 CRM 数据。
- 为用户提供“从模拟数据走向真实数据”的能力。

建议接口：

- `POST /api/import/customers`
- `POST /api/import/contacts`
- `POST /api/import/deals`
- `POST /api/import/follow-ups`

建议能力：

- CSV 模板下载。
- 字段校验。
- 重复数据处理。
- 导入预览。
- 导入结果报告。
- 错误行下载。

验收标准：

- 可以导入至少 50 条客户数据。
- 错误数据不会破坏已有数据。
- 导入后可立即参与风险扫描。

### 5.4 Alembic 迁移完善

优先级：高。

目标：

- 建立正式数据库迁移链路。
- 让项目更像真实工程项目。

建议任务：

- 生成首个 migration revision。
- 校验 migration 与 `init_schema.sql` 一致。
- 在 README 中补充数据库初始化方式。

验收标准：

- 新环境可以通过 Alembic 创建表结构。
- 也可以通过初始化 SQL 快速重建演示环境。

### 5.5 Docker Milvus 正式链路验证

优先级：高。

目标：

建议测试：

- 使用 `MILVUS_URI=http://localhost:19530`。
- 执行 RAG 入库。
- 执行 RAG 检索。
- 执行 RAG 评估。
- 检查 RAG Trace 和命中记录。

验收标准：

- Docker Milvus 下入库、检索、评估全部通过。

### 5.6 RQ + Redis + 前端按钮完整联调

优先级：高。

目标：

- 验证真实异步任务从前端按钮到 Worker 消费的完整链路。

测试链路：

```text
前端按钮
  -> FastAPI enqueue
  -> Redis queue
  -> RQ Worker consume
  -> MySQL / Milvus 落库
  -> 前端刷新查看结果
```

验收标准：

- 风险扫描按钮可用。
- 经营日报按钮可用。
- RAG 入库按钮可用。
- Worker 终端能看到任务执行日志。

## 6. 建议的下一版本开发顺序

建议按以下顺序推进：

1. 前端乱码与 UI 排版专项优化。
2. 真实 LangGraph 编排。
3. Docker Milvus 正式链路验证。
4. RQ + Redis + 前端按钮完整联调。
5. CSV 导入模块。
6. Alembic migration 完善。
7. 任务执行闭环。
8. 客户详情页。
9. RAG 文档管理页。
10. 报表增强。

这样安排的原因：

- 先修前端展示问题，可以立刻提升项目演示观感。
- 再补 LangGraph，可以提升“Agent 系统”的技术含金量。
- 然后做真实中间件联调，保证工程可信。
- 最后再补业务深度和产品完整度。

## 8. 总结

当前版本已经具备 InsightPilot 的主干能力，可以作为 V1 Demo 使用。

但如果目标是“简历里让 HR 眼前一亮”，下一步必须重点补两类内容：

- 第一类是观感：修复乱码、优化排版、增强前端展示。
- 第二类是技术深度：真实 LangGraph 编排、RAG 评估、异步任务完整联调、CSV 导入和 Alembic 迁移。

完成这些后，InsightPilot 会从“能跑通的 Demo”升级为“可解释、可审计、可扩展、可展示的企业级 Agent 系统”。
